# Контекстная карта (Context Map)

## 1 Обзор системы

MCP Odoo JSON-RPC — сервер протокола Model Context Protocol,
предоставляющий AI-ассистентам доступ к управлению задачами
в Odoo ERP через внутренний JSON-RPC протокол.

**Целевая версия Odoo**: 17 (использует `specification` dict вместо `fields` list в Odoo <= 16).

## 2 Ограниченные контексты (Bounded Contexts)

### 2.1 Odoo Project Management (upstream)

**Владелец**: Odoo ERP (внешняя система)
**Тип**: Big Ball of Mud (монолитная ERP с единой моделью данных)

Содержит:

| Модель Odoo          | Описание                        |
|----------------------|---------------------------------|
| `project.task`       | Задачи проекта                  |
| `project.project`    | Проекты                         |
| `res.users`          | Пользователи                    |
| `res.partner`        | Контакты / партнёры             |
| `account.analytic.line` | Трудозатраты (timesheets)    |
| `mail.message`       | Сообщения и история изменений   |
| `project.task.type`  | Стадии задач (Kanban-колонки)   |

**Характеристики**:
- Не предоставляет публичного API
- Использует внутренний JSON-RPC протокол для веб-клиента
- Модель данных включает ORM-специфичные конструкции
- Возвращает `false` вместо `null`/`None`

### 2.2 MCP Task Management (downstream)

**Владелец**: команда разработки MCP-сервера
**Тип**: чистый домен с явными границами

Содержит:

| Доменный объект | Источник Odoo          | Описание                    |
|-----------------|------------------------|-----------------------------|
| `Task`          | `project.task`         | Задача с основными полями   |
| `Project`       | `project.project`      | Проект (id + название)      |
| `User`          | `res.users`            | Пользователь Odoo           |
| `Stage`         | `project.task.type`    | Стадия задачи               |
| `Message`       | `mail.message`         | Сообщение / комментарий     |
| `Timesheet`     | `account.analytic.line`| Запись трудозатрат          |
| `Tag`           | `project.tags`         | Тег задачи                  |

**Характеристики**:
- Чистая доменная модель без Odoo-специфики
- Значения нормализованы (`None` вместо `false`)
- Идентификаторы и связи представлены явно

**Архитектурная особенность Odoo 17**:
Timesheets (`account.analytic.line`) не имеют самостоятельного CRUD-эндпоинта.
Создание, чтение и обновление трудозатрат происходит исключительно через
родительскую запись `project.task/web_save` с X2M-командами в поле `timesheet_ids`.
Это означает, что в ACL Protocol Layer операции с трудозатратами реализуются
как модификация X2M-поля задачи, а не как отдельные вызовы.

## 4 Классификация отношений

### 4.1 Odoo → MCP Server: Conformist

| Характеристика                 | Значение                              |
|--------------------------------|---------------------------------------|
| Направление                    | Odoo (upstream) → MCP (downstream)    |
| Тип отношения                  | Conformist                            |
| Переговорная позиция           | Отсутствует                           |
| Стабильность контракта         | Не гарантирована                      |
| Документация протокола         | Отсутствует (внутренний протокол)     |

**Почему не Open Host Service (OHS)**:
OHS предполагает, что upstream *явно публикует* протокол для внешних
потребителей с гарантиями обратной совместимости. JSON-RPC эндпоинты
Odoo — внутренний механизм веб-клиента, не предназначенный для
внешнего использования.

**Почему не Customer-Supplier**:
Customer-Supplier предполагает, что upstream *учитывает потребности*
downstream при развитии протокола. Odoo не знает о существовании
нашего MCP-сервера.

### 4.2 MCP Server → AI-ассистент: Open Host Service

| Характеристика                 | Значение                              |
|--------------------------------|---------------------------------------|
| Направление                    | MCP Server (upstream) → AI (downstream)|
| Тип отношения                  | Open Host Service + Published Language|
| Протокол                       | Model Context Protocol (MCP)          |
| Стабильность                   | Гарантирована спецификацией MCP       |

MCP-сервер сам выступает как OHS для AI-ассистентов, предоставляя
стабильный набор инструментов через стандартизированный протокол.

## 5 Dev-mode: контекст разведки протокола

### 5.1 Назначение

Отдельный вспомогательный контекст, не входящий в runtime MCP-сервера.
Предназначен исключительно для разработчиков: обнаружение эндпоинтов,
захват структур запросов/ответов, маппинг значений полей.

### 5.2 Расположение в архитектуре

```
 ┌──────────────────────────────────────────────────────────┐
 │                    Odoo ERP                              │
 └──────────┬───────────────────────────┬───────────────────┘
            │                           │
            │ JSON-RPC                  │ JSON-RPC
            │ (runtime)                 │ (capture)
            │                           │
            ▼                           ▼
 ┌──────────────────┐       ┌───────────────────────────┐
 │   MCP Server     │       │   Dev-mode (Playwright)   │
 │   (production)   │       │   Перехват трафика →      │
 │                  │  ◄──  │   генерация структур →    │
 │   ACL, Tools     │ питает│   docs/api/ обновление    │
 └──────────────────┘       └───────────────────────────┘
```

Dev-mode **питает** MCP-сервер знаниями о протоколе:
- Обнаруженные эндпоинты → `docs/api/`
- Структуры запросов → ACL Protocol Layer
- Значения enum-полей → Domain Translation Layer
- Error-responses → обработка ошибок в ACL

### 5.3 Границы контекста

| Аспект              | Dev-mode                  | MCP Server (runtime)     |
|---------------------|---------------------------|--------------------------|
| Пользователь        | Разработчик MCP-сервера   | AI-ассистент             |
| Зависимости         | Playwright, Chromium      | httpx, mcp SDK           |
| Артефакты           | JSON-структуры, отчёты    | MCP tool responses       |
| Поставка            | Не входит в пакет         | pip install              |
| Open-source         | Нет (внутренний)          | Да (планируется)         |

## 6 Стратегия защиты от изменений

### 6.1 Версионирование совместимости

MCP-сервер привязан к конкретной версии Odoo. При обновлении Odoo
необходимо:

1. Проверить работоспособность всех JSON-RPC эндпоинтов
2. Обновить ACL при изменении структуры запросов/ответов
3. Доменная модель и MCP-инструменты не должны изменяться

### 6.2 Обнаружение несовместимостей

- Интеграционные тесты против реального инстанса Odoo
- Валидация структуры ответов в Protocol Layer
- Явные ошибки при неожиданных форматах данных

### 6.3 Dev-mode как инструмент адаптации

При обновлении Odoo:

1. Запустить `dev/capture.py` на новой версии
2. Сравнить `dev/captured/endpoints.json` с предыдущей версией
3. Обнаруженные расхождения → обновить ACL
4. Доменная модель и MCP-инструменты не должны изменяться

## 7 Поток данных

```
AI-ассистент
    │
    │ MCP: tool_call("list_tasks", {...})
    ▼
MCP Server (list_tasks tool)
    │
    │ Domain: TaskRepository.find_by_user(uid)
    ▼
ACL: Domain Translation
    │
    │ Формирует specification, domain filter
    ▼
ACL: Protocol Layer
    │
    │ JSON-RPC envelope + params
    ▼
ACL: Transport Layer
    │
    │ POST /web/dataset/call_kw/project.task/web_search_read
    │ Cookie: session_id=...
    ▼
Odoo ERP
    │
    │ JSON-RPC response
    ▼
ACL: Protocol Layer
    │
    │ Извлекает result.records
    ▼
ACL: Domain Translation
    │
    │ OdooRecord → Task (нормализация)
    ▼
MCP Server
    │
    │ MCP: tool_result (список задач)
    ▼
AI-ассистент
```
