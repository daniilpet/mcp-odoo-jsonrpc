# ADR-002: Стратегия аутентификации

## Статус

Accepted

## Контекст

Odoo использует сессионную аутентификацию через cookie `session_id`.
Cookie имеет флаг `HttpOnly` и привязан к домену инстанса.

Для взаимодействия с Odoo JSON-RPC необходимо передавать валидный
`session_id` в каждом HTTP-запросе.

Существует два подхода к получению сессии:

1. Логин через `/web/session/authenticate` (login + password)
2. Использование заранее полученного `session_id`

### Ограничения

- MCP-сервер работает как фоновый процесс без UI
- Нет возможности интерактивного ввода учётных данных
- `session_id` имеет ограниченное время жизни
- Odoo может использовать 2FA, LDAP или другие схемы аутентификации

## Рассмотренные варианты

### Вариант A: session_id из переменной окружения

- **Плюсы**:
  - Простота реализации
  - Не требует хранения логина/пароля
  - Работает с любой схемой аутентификации Odoo (2FA, LDAP, OAuth)
  - Пользователь полностью контролирует процесс авторизации

- **Минусы**:
  - Требует ручного обновления при истечении сессии
  - Пользователь должен знать, где найти `session_id` в браузере

### Вариант B: Логин через `/web/session/authenticate`

- **Плюсы**:
  - Автоматическое обновление сессии
  - Не требует ручных действий пользователя

- **Минусы**:
  - Хранение логина и пароля в ENV — риск безопасности
  - Не работает с 2FA, LDAP, OAuth и другими нестандартными схемами
  - Усложняет конфигурацию (больше переменных окружения)
  - Требует реализации логики переавторизации

### Вариант C: Гибридный (session_id + fallback на логин)

- **Плюсы**:
  - Гибкость — работает в обоих режимах

- **Минусы**:
  - Сложность реализации и тестирования
  - Два пути аутентификации — два источника ошибок
  - Избыточность для MVP

## Решение

Выбран **Вариант A: session_id из переменной окружения**.

### Обоснование

1. Работает с любой конфигурацией аутентификации Odoo
2. Не требует хранения учётных данных
3. Минимальная поверхность атаки
4. Простота реализации для MVP
5. Документация объясняет процесс получения `session_id`

### Валидация сессии

При старте MCP-сервер выполняет проверочный запрос
`res.users/search_read` для валидации `session_id`.
При невалидной сессии — явная ошибка с инструкцией по обновлению.

## Последствия

- **Положительные**: безопасность, универсальность, простота
- **Отрицательные**: ручное обновление session_id при истечении
- **Риски**: частое истечение сессии может снижать UX
- **Митигация**: добавить понятное сообщение об ошибке с инструкцией
