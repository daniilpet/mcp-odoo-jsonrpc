# ADR-007: Отказ от универсального execute tool

## Статус

Accepted

## Контекст

Конкурирующие MCP-серверы для Odoo (angelmoya/mcp-odoorpc) предоставляют
универсальный `execute` tool, позволяющий вызвать любой метод любой модели
Odoo, а также MCP Resources для произвольного browse записей
(`odoo://record/{model}/{id}`).

Вопрос: добавлять ли аналогичную функциональность?

## Рассмотренные варианты

### Вариант A: Добавить универсальный execute

- **Плюсы**:
  - Работает с любой моделью Odoo (CRM, Sales, HR, и пр.)
  - Не требует разработки специализированных инструментов
  - Паритет с конкурентами

- **Минусы**:
  - AI не знает структуру ответа — результат непредсказуем
  - Невозможно применить security model (restricted/full):
    нельзя фильтровать поля, если не знаешь какая модель запрашивается
  - Нет типизации и валидации — ошибки обнаруживаются только в runtime
  - Нарушает принцип Anti-Corruption Layer: Odoo ORM протекает к AI
  - Поверхность атаки: AI может вызвать `unlink`, `write` на произвольной модели

### Вариант B: Не добавлять (текущее решение)

- **Плюсы**:
  - Предсказуемый результат каждого инструмента
  - Security model работает: restricted specs не запрашивают sensitive данные
  - Типизированные доменные объекты (Pydantic v2)
  - ACL изолирует Odoo-специфику от потребителей
  - Контролируемая поверхность атаки

- **Минусы**:
  - Ограниченный scope: только задачи + timesheets
  - Для новых модулей нужна отдельная разработка

## Решение

Выбран **Вариант B: не добавлять универсальный execute**.

### Обоснование

Это осознанное архитектурное решение, а не ограничение:

1. **Предсказуемость > универсальность**: каждый инструмент возвращает
   типизированный результат с известной структурой
2. **Безопасность**: restricted mode невозможен для произвольных запросов
3. **Качество**: специализированные инструменты дают лучший UX для AI
4. **Принцип ACL**: Odoo ORM не должен протекать к потребителям

### Расширение scope

При необходимости поддержки новых модулей Odoo (CRM, Sales и пр.)
правильный путь:
1. Добавить специализированные инструменты с типизацией
2. Расширить доменную модель
3. Добавить restricted specs для нового модуля
4. Покрыть тестами

## Последствия

- **Положительные**: безопасность, предсказуемость, типизация
- **Отрицательные**: ограниченный scope, требует разработки для новых модулей
- **Конкурентные**: позиционирование как "безопасный и предсказуемый"
  vs "универсальный, но без гарантий"
