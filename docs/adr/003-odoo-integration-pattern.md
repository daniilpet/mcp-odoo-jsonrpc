# ADR-003: Паттерн интеграции с Odoo

## Статус

Accepted

## Контекст

MCP-сервер является downstream-потребителем данных из Odoo ERP.
Необходимо определить стратегический DDD-паттерн для описания
отношений между двумя системами.

### Характеристики upstream-системы (Odoo)

1. **Отсутствие публичного API**: Odoo не предоставляет документированного,
   версионированного REST/GraphQL API для внешних потребителей
2. **Внутренний протокол**: JSON-RPC эндпоинты (`/web/dataset/call_kw/*`)
   являются внутренним протоколом веб-клиента Odoo
3. **Нестабильность контракта**: структура запросов и ответов может измениться
   при обновлении Odoo без предварительного уведомления
4. **Отсутствие переговорной позиции**: Odoo не знает о существовании
   нашего MCP-сервера и не будет адаптировать свой протокол

### Характеристики downstream-системы (MCP-сервер)

1. Предоставляет стабильный набор MCP-инструментов для AI-ассистентов
2. Должен изолировать потребителей от нестабильности upstream
3. Должен транслировать Odoo-специфичные концепции в понятную модель

## Рассмотренные варианты

### Вариант A: Open Host Service (OHS) + Published Language

- **Отклонён**: OHS предполагает, что upstream **явно публикует**
  версионированный протокол для внешних потребителей. JSON-RPC Odoo —
  внутренний протокол веб-клиента, а не опубликованный сервис.

### Вариант B: Customer-Supplier

- **Отклонён**: Customer-Supplier предполагает наличие **переговорной
  позиции** — upstream учитывает потребности downstream. Odoo не знает
  о нашем существовании и не будет адаптировать протокол.

### Вариант C: Conformist (без ACL)

- **Отклонён частично**: Conformist верно описывает отношение с upstream,
  но без Anti-Corruption Layer внутренние структуры Odoo (ORM-команды,
  доменные фильтры, формат `false` вместо `null`) протекут в доменную
  модель MCP-сервера.

### Вариант D: Conformist + Anti-Corruption Layer

- **Принят**: точно описывает реальность — мы подчиняемся протоколу Odoo
  (Conformist), но защищаем свою доменную модель через ACL.

## Решение

Выбран паттерн **Conformist + Anti-Corruption Layer (ACL)**.

### Обоснование

| Критерий                       | Наш случай                                    |
|--------------------------------|-----------------------------------------------|
| Upstream публикует API?        | Нет — внутренний протокол                     |
| Upstream учитывает downstream? | Нет — не знает о нас                          |
| Контракт стабилен?             | Нет — может измениться при обновлении          |
| Модели совместимы?             | Нет — ORM-команды, `false` вместо `null`      |

Все четыре фактора указывают на Conformist + ACL.

### Архитектура Anti-Corruption Layer

ACL состоит из трёх слоёв:

```
┌─────────────────────────────────────────────┐
│              MCP Tools Layer                │
│  (list_tasks, create_task, log_timesheet)   │
├─────────────────────────────────────────────┤
│         Domain Translation Layer            │
│  OdooTask → Task, OdooTimesheet → Timesheet │
│  false → None, [4, id] → link semantics     │
├─────────────────────────────────────────────┤
│           Protocol Layer                    │
│  JSON-RPC serialization/deserialization     │
│  Error mapping: Odoo errors → exceptions    │
│  Request ID management                      │
├─────────────────────────────────────────────┤
│           Transport Layer                   │
│  HTTP client (httpx)                        │
│  session_id cookie management               │
│  Retry logic, timeout handling              │
├─────────────────────────────────────────────┤
│                                             │
│           Odoo ERP (upstream)               │
│         JSON-RPC internal protocol          │
│                                             │
└─────────────────────────────────────────────┘
```

#### Transport Layer (Транспортный слой)

- HTTP-клиент с управлением cookies
- Передача `session_id` в каждом запросе
- Retry-политика и таймауты
- Обработка сетевых ошибок

#### Protocol Layer (Протокольный слой)

- Формирование JSON-RPC envelope (`id`, `jsonrpc`, `method`, `params`)
- Сериализация Odoo-специфичных конструкций:
  - Domain-фильтры: `["&", ["field", "op", value]]`
  - Many2many команды: `[4, id]`, `[3, id]`, `[6, 0, [ids]]`
  - Specification-объекты для выбора полей
- Десериализация ответов и маппинг ошибок в исключения

#### Domain Translation Layer (Слой трансляции домена)

- Преобразование Odoo-записей в доменные объекты MCP-сервера
- Нормализация значений: `false` → `None`
- Маппинг идентификаторов: `{"id": 99, "display_name": "Example Project"}` → `Project(id=99, name="Example Project")`
- Скрытие внутренних деталей Odoo от MCP-инструментов

## Последствия

- **Положительные**:
  - Изменения в протоколе Odoo локализуются в ACL
  - MCP-инструменты работают с чистой доменной моделью
  - Возможность тестирования без реального Odoo (mock ACL)
  - При обновлении Odoo нужно адаптировать только ACL

- **Отрицательные**:
  - Дополнительный слой абстракции увеличивает объём кода
  - Необходимость поддерживать маппинг при каждом изменении Odoo

- **Риски**:
  - Обновление Odoo может сломать JSON-RPC протокол без предупреждения
  - Митигация: версионирование совместимости, интеграционные тесты
    против конкретной версии Odoo
