# ADR-005: Модель безопасности

## Статус

Accepted

## Контекст

MCP-сервер подключает AI-ассистента к корпоративной ERP-системе.
Это создаёт несколько векторов угроз:

### Угрозы

| Вектор                           | Описание                                              | Критичность |
|----------------------------------|-------------------------------------------------------|-------------|
| Утечка session_id                | Plaintext в ENV/файле → полный доступ к Odoo          | Критическая |
| Утечка данных через AI-контекст  | Названия задач, описания, имена → отправляются в LLM  | Высокая     |
| Неавторизованная модификация     | AI может создать/изменить задачи без явного намерения  | Средняя     |
| Публичный репозиторий            | Снижает барьер для злоупотребления инструментом        | Низкая      |

### Ключевой конфликт

Полезность MCP-сервера прямо зависит от объёма данных, передаваемых
AI-ассистенту. Чем больше контекста — тем полезнее ответы. Но чем
больше данных в контексте LLM — тем выше риск утечки корпоративной
информации.

## Рассмотренные варианты

### Вариант A: Маскирование данных (****)

- **Отклонён**: замена части данных на `***` неэффективна.
  AI не может работать с замаскированными данными, а при
  частичной маскировке утечка всё равно возможна по контексту.

### Вариант B: Двухуровневый режим доступа (restricted / full)

- **Принят**: по умолчанию MCP работает в ограниченном режиме,
  показывая только структурные данные без контента. Полный доступ
  включается явным флагом доверия.

### Вариант C: ACL на уровне полей

- **Отклонён как избыточный**: гранулярный контроль каждого поля
  усложняет конфигурацию и код без пропорционального выигрыша.
  Два уровня покрывают 95% сценариев.

## Решение

### 1. Двухуровневый режим доступа

#### Restricted Mode (по умолчанию)

AI-ассистент видит **только структурные метаданные**:

| Инструмент        | Restricted (по умолчанию)                    | Full (с флагом)              |
|-------------------|----------------------------------------------|------------------------------|
| `list_tasks`      | ID, stage, priority, deadline, hours         | + name, project, tags        |
| `get_task`        | ID, stage, hours, timesheets (без описаний)  | + name, description, messages, attachments |
| `create_task`     | Доступен (данные от пользователя)            | Без изменений                |
| `update_task`     | Доступен                                     | Без изменений                |
| `change_task_stage` | Доступен                                   | Без изменений                |
| `log_timesheet`   | Доступен (данные от пользователя)            | Без изменений                |
| `get_timesheets`  | Только часы и даты                           | + описания работ             |

**Логика**: данные, которые вводит сам пользователь (create_task,
log_timesheet) — не утечка. Утечка — это когда AI читает
корпоративные данные, созданные другими людьми.

#### Full Mode (явное включение)

Активируется через:
- ENV: `ODOO_TRUST_MODE=full`
- CLI: `mcp-odoo-cli --trust`
- MCP: при запуске сервера с переменной окружения

### 2. Фильтр по проектам

```
ODOO_ALLOWED_PROJECTS=99,101
```

Если задан — MCP-сервер видит только задачи указанных проектов.
Запросы к другим проектам отклоняются с ошибкой.

Если не задан — видны все проекты пользователя (с учётом режима доступа).

### 3. Безопасное хранение session_id

Вместо plaintext `session.json` — использование системного хранилища
секретов через библиотеку `keyring`:

| Платформа | Хранилище                              |
|-----------|----------------------------------------|
| Windows   | Windows Credential Manager             |
| macOS     | Keychain                               |
| Linux     | Secret Service (GNOME Keyring / KWallet) |

#### Структура хранения

- **Ключ**: `odoo-cli:{base_url}`
- **Значение**: `session_id`
- **Метаданные (файл)**: `session.json` без session_id
  (uid, employee_id, company_ids, lang, tz — не секретные)

#### Fallback

Если `keyring` недоступен (headless-сервер, контейнер):
- Файл `session.json` с session_id (как сейчас)
- Предупреждение при старте: "session_id хранится в plaintext"
- ENV переменная `ODOO_SESSION_ID` (для CI/CD)

### 4. Матрица состояний

| keyring доступен | ODOO_SESSION_ID в ENV | session.json существует | Действие                            |
|------------------|-----------------------|-------------------------|-------------------------------------|
| ✓                | ✗                     | ✓ (без session_id)     | session_id из keyring              |
| ✓                | ✓                     | —                       | session_id из ENV, warning          |
| ✗                | ✗                     | ✓ (с session_id)       | session_id из файла, warning        |
| ✗                | ✓                     | —                       | session_id из ENV                   |
| ✗                | ✗                     | ✗                       | Ошибка: выполните login             |
| ✓                | ✗                     | ✗                       | Ошибка: выполните login             |

## Последствия

- **Положительные**:
  - Безопасный режим по умолчанию — не нужно думать о рисках
  - Явное включение полного доступа — осознанное решение
  - session_id в системном хранилище — не утечёт через git/backup
  - Фильтр проектов — минимизация поверхности атаки

- **Отрицательные**:
  - Restricted mode снижает полезность AI-ассистента
  - `keyring` — дополнительная зависимость
  - Два режима — дополнительная сложность в коде

- **Риски**:
  - Пользователи могут всегда включать full mode, обесценивая защиту
  - Митигация: документация объясняет риски каждого режима
  - `keyring` может не работать в некоторых окружениях
  - Митигация: graceful fallback на файл с warning
