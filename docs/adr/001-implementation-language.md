# ADR-001: Выбор языка реализации

## Статус

Accepted

## Контекст

Необходимо выбрать язык программирования для реализации MCP-сервера,
обеспечивающего интеграцию с Odoo через JSON-RPC.

MCP SDK доступен для двух языков: TypeScript и Python. Оба варианта
поддерживаются официально и имеют зрелые реализации.

### Факторы выбора

- Технологический стек команды: основной опыт в Python (Odoo, Django, Flask)
- Odoo сам написан на Python — единая экосистема упрощает отладку
- Существующий MCP-сервер Todoist в проекте реализован на Python
- HTTP-взаимодействие: оба языка имеют зрелые HTTP-клиенты

## Рассмотренные варианты

### Вариант A: Python

- **Плюсы**:
  - Единая экосистема с Odoo (Python)
  - Опыт команды в Python
  - Консистентность с существующим MCP-сервером Todoist
  - Богатая экосистема для работы с JSON-RPC (`httpx`, `aiohttp`)
  - Возможность переиспользования типов данных Odoo

- **Минусы**:
  - Менее строгая типизация по сравнению с TypeScript
  - GIL ограничивает параллелизм (некритично для I/O-bound MCP)

### Вариант B: TypeScript

- **Плюсы**:
  - Строгая типизация из коробки
  - Большое сообщество MCP-разработчиков на TypeScript
  - Более зрелый MCP SDK (TypeScript SDK появился первым)

- **Минусы**:
  - Разная экосистема с Odoo
  - Нет опыта команды в TypeScript для серверных приложений
  - Несогласованность со стеком существующих MCP-серверов проекта

## Решение

Выбран **Python** как язык реализации.

### Обоснование

1. Единство экосистемы с целевой системой (Odoo — Python)
2. Консистентность с существующим MCP-сервером Todoist
3. Компетенции команды сосредоточены в Python
4. Для I/O-bound MCP-сервера ограничения GIL несущественны
5. `httpx` обеспечивает асинхронный HTTP-клиент с поддержкой cookies

## Последствия

- **Положительные**: единая экосистема, быстрый старт, переиспользование опыта
- **Отрицательные**: необходимость ручного описания типов через dataclasses/Pydantic
- **Риски**: Python MCP SDK может отставать по фичам от TypeScript SDK
