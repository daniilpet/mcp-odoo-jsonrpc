# ADR-004: Dev-mode — инструментарий разведки эндпоинтов

## Статус

Accepted

## Контекст

Odoo не предоставляет документации по JSON-RPC протоколу.
Единственный способ узнать структуру запросов и ответов —
перехват трафика между веб-клиентом Odoo и сервером.

Ручной процесс (DevTools → копирование → форматирование) трудоёмок,
подвержен ошибкам и плохо воспроизводим. При обновлении Odoo или
добавлении новых инструментов в MCP-сервер процесс придётся повторять.

### Стратегический контекст

Текущий MCP-сервер создаётся для нужд команды. В перспективе
планируется публичный open-source репозиторий для произвольных
инстансов Odoo. Dev-mode инструментарий останется внутренним
инструментом разработчиков, не предназначенным для конечных
пользователей.

## Рассмотренные варианты

### Вариант A: Ручной перехват через DevTools

- **Плюсы**: нулевая стоимость разработки
- **Минусы**: ручной труд, ошибки копирования, невоспроизводимость,
  не масштабируется при добавлении новых эндпоинтов

### Вариант B: HTTP-прокси (mitmproxy)

- **Плюсы**: перехватывает весь трафик, зрелый инструмент
- **Минусы**: требует настройки сертификатов, внешняя зависимость,
  перехватывает лишний трафик (статика, WebSocket и пр.)

### Вариант C: Playwright с перехватом запросов

- **Плюсы**:
  - Запускает управляемый Chromium из Python
  - Перехватывает `fetch`/`XHR` на уровне страницы
  - Фильтрация JSON-RPC запросов по URL-паттернам
  - Автоматическая генерация структур после закрытия браузера
  - Единая экосистема Python (консистентность с MCP-сервером)
  - Пользователь работает в привычном интерфейсе Odoo

- **Минусы**:
  - Зависимость от Playwright (dev dependency)
  - Требует установки Chromium (~200 MB)

## Решение

Выбран **Вариант C: Playwright с перехватом запросов**.

### Обоснование

1. Пользователь работает в реальном Odoo — нет искусственных ограничений
2. Скрипт перехватывает только JSON-RPC (`/web/dataset/call_kw/*`,
   `/mail/*`) — без мусорного трафика
3. После закрытия браузера — автоматическая генерация:
   - Структур запросов с примерами
   - Структур ответов с типами полей
   - Обнаруженных значений enum-полей (state, priority)
   - Error-responses, если встретились
4. Выход — готовые файлы для пополнения `docs/api/`

### Сценарий использования

```
1. Разработчик запускает: python dev/capture.py
2. Открывается Chromium с Odoo (session_id из ENV)
3. Разработчик выполняет нужные действия в Odoo:
   - Открывает список задач (перехватывается web_search_read)
   - Создаёт задачу (перехватывается web_save)
   - Меняет стадию (перехватывается web_save + stage_id)
   - Списывает время (перехватывается analytic.line)
4. Разработчик закрывает браузер
5. Скрипт генерирует:
   dev/captured/
   ├── endpoints.json        — все перехваченные пары request/response
   ├── models.json           — обнаруженные модели и их поля
   └── report.md             — человекочитаемый отчёт
```

### Расположение

```
dev/
├── capture.py              — основной скрипт перехвата
├── analyze.py              — анализ и генерация структур
├── captured/               — результаты перехвата (gitignored)
│   ├── endpoints.json
│   ├── models.json
│   └── report.md
└── README.md               — инструкция для разработчиков
```

Папка `dev/` не входит в пакет MCP-сервера и не нужна конечным
пользователям.

## Последствия

- **Положительные**:
  - Воспроизводимый процесс обнаружения эндпоинтов
  - Быстрая адаптация при обновлении Odoo
  - Автоматическое обнаружение новых полей и значений
  - Документация генерируется из реальных данных

- **Отрицательные**:
  - Dev-зависимость от Playwright
  - Chromium занимает ~200 MB дискового пространства

- **Риски**:
  - Перехваченные данные могут содержать чувствительную информацию
  - Митигация: `dev/captured/` в `.gitignore`, предупреждение в README
