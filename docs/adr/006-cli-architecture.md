# ADR-006: Архитектура CLI — прямой доступ к ACL vs MCP-клиент

## Статус

Accepted

## Контекст

В проекте два entry point: MCP-сервер (для AI) и CLI (для человека).
Оба взаимодействуют с Odoo. Вопрос: как CLI получает данные?

## Рассмотренные варианты

### Вариант A: CLI → ACL напрямую (текущая реализация)

```
Odoo ERP
   ↑
  ACL (transport → protocol → mapper)
   ↑              ↑
MCP Server     CLI
   ↑              ↑
AI-ассистент   Человек
```

CLI и MCP-сервер — два независимых клиента одного ACL.

- **Плюсы**:
  - Простота: CLI не зависит от запущенного MCP-сервера
  - Производительность: нет промежуточного слоя
  - CLI работает автономно — достаточно `pip install`
  - Гибкость: CLI может использовать full mode независимо от MCP
  - Меньше точек отказа

- **Минусы**:
  - Два потребителя ACL — дублирование форматирования (server.py + cli.py)
  - Security model применяется по-разному (CLI всегда full, MCP — restricted)
  - При добавлении нового инструмента — обновлять и MCP, и CLI
  - Нет единой точки аудита: кто что запрашивал

### Вариант B: CLI → MCP Server (CLI как MCP-клиент)

```
Odoo ERP
   ↑
  ACL
   ↑
MCP Server
   ↑         ↑
AI-ассистент  CLI (MCP-клиент)
```

CLI подключается к MCP-серверу как клиент (stdio или SSE).

- **Плюсы**:
  - Единая точка входа — одна security policy для всех
  - Единая точка аудита: все запросы через MCP-сервер
  - DRY: новый инструмент добавляется один раз
  - CLI автоматически получает все MCP-tools без дублирования
  - Архитектурная чистота: один сервер, много клиентов

- **Минусы**:
  - CLI зависит от MCP-сервера (нужно запустить или встроить)
  - Дополнительная латентность: CLI → MCP → ACL → Odoo (вместо CLI → ACL → Odoo)
  - CLI ограничен интерфейсом MCP-инструментов (только string in → string out)
  - Потеря rich-форматирования: MCP-tools возвращают текст, CLI должен его парсить
    или дублировать форматирование поверх
  - Сложность запуска: CLI должен поднять MCP-сервер как subprocess
    или подключиться к уже запущенному
  - Security model MCP (restricted) конфликтует с потребностями CLI (full):
    CLI пришлось бы переключать режим или обходить ограничения
  - MCP протокол не предназначен для rich TUI — он заточен под AI-ассистентов

### Вариант C: Гибрид — общий Service Layer

```
Odoo ERP
   ↑
  ACL
   ↑
Service Layer (бизнес-логика: list_tasks, get_task, log_timesheet)
   ↑              ↑
MCP Server     CLI
(форматирует   (форматирует
 для AI)        для терминала)
```

Общий слой сервисов между ACL и представлениями.

- **Плюсы**:
  - DRY бизнес-логика — добавляется один раз в Service Layer
  - CLI и MCP — тонкие адаптеры форматирования
  - CLI не зависит от MCP-сервера
  - Каждый адаптер форматирует под свою аудиторию
  - Security model в Service Layer — единая для всех
  - Тестируемость: Service Layer тестируется отдельно от представлений

- **Минусы**:
  - Дополнительный слой абстракции
  - Текущий server.py придётся рефакторить: вынести логику из MCP-tools
  - Для 7 инструментов может быть избыточно

## Сравнительная таблица

| Критерий                    | A: CLI → ACL | B: CLI → MCP | C: Service Layer |
|-----------------------------|--------------|--------------|------------------|
| Простота запуска CLI        | +++          | +            | ++               |
| DRY (нет дублирования)     | +            | +++          | +++              |
| Rich TUI форматирование    | +++          | +            | +++              |
| Единая security policy     | +            | +++          | ++               |
| Производительность          | +++          | +            | ++               |
| Автономность CLI            | +++          | +            | +++              |
| Масштабируемость            | +            | ++           | +++              |
| Объём рефакторинга          | 0            | +++          | ++               |

## Решение

Выбран **Вариант C: гибрид с общим Service Layer**.

### Обоснование

1. DRY — бизнес-логика в одном месте
2. CLI и MCP — тонкие адаптеры: MCP форматирует строки для AI, CLI использует rich для терминала
3. CLI не зависит от запущенного MCP-сервера
4. Security model в Service Layer — единая для всех потребителей
5. Масштабируемость: при добавлении нового инструмента — один метод в Service Layer

### План миграции

Текущая реализация (Вариант A) работает. Миграция на C при расширении
функционала — вынос логики из `server.py` в `service.py`.

## Последствия

- **Положительные**: DRY, тестируемость, единая security policy
- **Отрицательные**: дополнительный слой абстракции
- **Риски**: over-engineering для 7 инструментов
- **Митигация**: миграция поэтапно, при расширении scope
